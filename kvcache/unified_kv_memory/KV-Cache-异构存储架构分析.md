# 统一异构 KV Cache 存储集群架构与分层需求规格分析报告

## 一、 第一性原理分析：KV Cache 的核心本质与多级存储的必要性辨析

在大语言模型（LLM）的在线推理（Serving）中，**KV Cache 的本质是大模型自回归解码（Autoregressive Decode）过程中，空间换时间的阶段性激活态（Activation）数据**。

从第一性原理出发，评估 KV Cache 的存储、卸载（Offload）与加载（Reload）的正确性、完整性和必要性，必须回归到**算力（Compute Bound）与带宽（Memory Bound）的经典平衡模型**。

### 1. 辩证思考：重算（Recomputation）与传输（Transfer）的效能边界
传统的观点倾向于“尽可能保留和复用 KV Cache”。但从定量科学的角度来看，这一设想是否总是成立？
我们以一个 $70\text{B}$ 参数模型（如 Llama-3-70B，采用 GQA 架构：80层，8个 KV 头，128项头维度，FP16 精度）在单张英伟达 B200 显卡上的表现为例：
* **单 Token 的 KV Cache 大小**：
    $$2 \times 80 \times 8 \times 128 \times 2 \text{ Bytes} = 327,680 \text{ Bytes} \approx 320 \text{ KB/Token}$$
* **计算开销（Prefill 阶段）**：
    不考虑 $O(N^2)$ 的 Attention 矩阵计算，仅矩阵乘法（GEMM）部分，每处理一个 Token 约消耗 $2 \times 70\text{B} = 140 \text{ GFLOPs}$ 的算力。若显卡实际 Prefill 算力发挥为 $500 \text{ TFLOPs}$，则计算 1024 个 Token 的 Prefill 时间约为：
    $$\frac{1024 \times 140 \text{ GFLOPs}}{500 \text{ TFLOPs}} \approx 286.7 \text{ ms}$$
* **传输开销（PCIe Gen5 $\times 16$ 链路，单向约 $64 \text{ GB/s}$）**：
    1024 个 Token 的 KV Cache 总量为 $327.68 \text{ MB}$。将其从本地主机内存（DDR）加载到显存（HBM）的时间约为：
    $$\frac{327.68 \text{ MB}}{64 \text{ GB/s}} \approx 5.12 \text{ ms}$$

> **科学结论**：
> 在 $1024 \text{ Tokens}$ 规模下，**传输耗时（$5.12\text{ ms}$）远小于重算耗时（$286.7\text{ ms}$）**。随着 Context 长度增加，重算的算力消耗呈 $O(N^2)$ 级增长，而 KV Cache 大小仅呈 $O(N)$ 线性增长。
> 
> 因此，构建多级异构存储进行 KV Cache 的卸载与加载，在物理法则上具有**绝对的必要性与收益合理性**。特别是在长文本（Long-Context）、多轮对话、RAG（检索增强生成）以及 Prefill/Decode 算力分离（Disaggregated P/D）的场景下，其经济效益与性能收益极高。

用户描述中提到：“由于每次传输的多个 block 不连续，因此采用 for循环多次调用传输的方式”。**从框架底层事实来看，这一描述存在完整性与准确性偏差。** 现代高性能推理框架（如 vLLM 和 SGLang）为了规避 Python 层面的 `for` 循环带来的 CPU Launch Overhead 及 PCIe 碎片化流产，已经演进出了更为高级的向量化与异步批处理机制。

## 四、 推演场景：多级异构介质（GPU/NPU, 本地/远端 DDR, 本地/远端 SSD）的 KV Cache 协同流动

### 1. 软硬件性能指标诉求矩阵（2026 最新技术栈定量基准）

### 2. 深度推演：跨介质流动三大核心业务痛点分析

#### 痛点一：多流并发导致的“流量风暴”（Traffic Storm）
* **现象描述**：在超大规模集群（数千张卡）执行长文本 Serving 时，若遇到全网大批量 Prefill 请求同时结束，传输管理层会瞬间下发海量的 KV Cache 远端备份任务。此时，跨机 RoCEv2 网络上会出现严重冲突，瞬间打满 800Gbps 交换机背板，造成重传丢包，导致关键 Decode 节点的吞吐率断崖式下跌。
* **第一性原理剖析**：由于缺乏全局拓扑感知和网络拥塞流控（PFC/ECN 在极高突发下失效），非连续小块并发请求直接造成线头阻塞（Head-of-Line Blocking）。

#### 痛点二：传输总线（PCIe）与算子下发及多模态前后处理的“业务阻塞”
* **现象描述**：在单节点内，当异步线程通过 PCIe 执行 KV Cache 卸载（HBM $\rightarrow$ DDR）时，上层框架正在高频下发 Decode 阶段的 CUDA Kernel。此时发现，显卡前端的命令处理器（Command Processor）发生挂起，算子延迟飙升，TPOT 产生高达 $50\text{-}100\text{ ms}$ 的剧烈抖动（Jitter）。
* **第一性原理剖析**：传统的 `cudaMemcpyAsync` 虽然是异步的，但频繁的非连续小块物理地址页映射会长时间霸占 **PCIe 控制器的 DMA 引擎与 MMU 映射表**，导致 GPU 的硬件命令队列（Hardware Command Queue）和 Host-to-Device 的控制流信号（Kernel Launch）发生物理排队冲突。

#### 痛点三：跨多层非连续块碎片化（Fragmentation）导致传输效率黑洞
* **现象描述**：经过数小时的连续运行，显存和系统内存都被大量离散的 16-Token 块切碎。当下达一个传输 50MB KV Cache 的指令时，由于这些块在物理上极其零散，底层 DMA 引擎被迫将其拆解为数万个极小的 PCIe 包（TLP Packet），使得总线实际吞吐效率跌至理论值的 $10\%$ 以下。

### 3. 高性能未来架构演进方向：统一异构 KV Cache 存储集群

为了彻底解决上述痛点，未来的软硬件架构必须朝着**显式拓扑自适应、计算传输物理分离、以及全局虚拟池化**三个方向演进。

```
     [全局虚拟统一内存空间 (Unified Global Memory Space)]
 ┌──────────────────────┬──────────────────────┬──────────────────────┐
 │       HBM 池         │      CXL DDR 池      │     远端 RDMA 池     │
 └──────────────────────┴──────────────────────┴──────────────────────┘
    ▲                      ▲                      ▲
    │ 硬件级动态路由映射     │ CXL 3.0 直接内存访问   │ 800G 零拷贝直接直连
    └──────────────────────┴──────────────────────┴──────────────────────┘
                  [拓扑感知与拥塞避免感知流控内核] (全局感知)
```

* **技术突破一：面向统一内存（Unified Memory）与 CXL 3.0 的硬件级透明路由**
    未来架构应彻底抛弃高层显式调用的 Swapping 算子。借助 CXL 3.0 提供的 **Shared Memory 协议**，GPU 的地址空间与主机的 DDR 甚至远端节点的内存直接构建在一个 **全局虚拟内存空间（Unified Global Memory Space）** 中。
    当显存不足时，硬件 MMU 自动将冷页的物理映射变更为 CXL 链路对端，算子读取时由硬件控制器实现自动按需拉取（Hardware Page Fault-free Pre-fetching），数据流动对框架完全透明，速度可达数百 GB/s。
* **技术突破二：动态网络拥塞感知与拓扑自适应调度（Topology-aware Scheduling）**
    传输管理层（L3）必须内嵌一个**软硬件协同的感知拓扑拓扑引擎**。在执行 KV Cache 远端传输时，优先匹配同一 NVLink 交换机组内的节点；其次通过 PCIe 树状局部近邻转发；网络传输时，感知当前 800G 网卡的拥塞标志位（ECN），动态压制备份流量的带宽上限，确保高优先级的 Decode 网络流绝对顺畅。

## 一、 关键控制路径的痛点剖析与软硬件方案增强

### 1. 前缀索引计算（Prefix Indexing）
* **现状痛点**：当前 vLLM/SGLang 普遍在 CPU 端利用 Python/C++ 对 Token ID 序列进行哈希计算（如 `hash(tuple(token_ids))`）。当 Prompt 长度达到 32K 甚至 128K 以上时，CPU 串行扫描数万个整型数据并生成多级 Rolling Hash，其时延在并发高时会突破 $5\text{-}10\text{ ms}$。
* **硬件增强**：利用 **CPU 向量化指令集（AVX-512 / ARM Neon）** 或直接在网卡/DPU 内置的 **哈希硬件加速引擎**。
* **软件方案**：**分块并行增量哈希（Chunked Incremental Hashing）**。API Gateway 在接收 Prompt 阶段，每流式接收到一个 Chunk（如 1024 Token）就立即在背景线程计算该段的独立 Hash 并上报，彻底将哈希计算时延与请求排队、Tokenizer 耗时隐蔽重叠。

### 2. 前缀匹配上的 KV Cache 查找（Match Lookup）
* **现状痛点**：单机内 Radix Tree 的深度遍历需要频繁加锁（Lock Contention），而在分布式集群中（如分布式 P/D 分离架构），去中心化或集中式的元数据服务（元数据存储在 Redis 或 Mooncake Master）面临着跨网络 RPC 的往返时延（RTT，约 $50\text{-}100\mu\text{s}$），在高并发下极易形成控制流的排队风暴。
* **硬件增强**：**CXL 3.0 内存池化（Fabric-Attached Memory）**。将全局 KV Cache 索引树直接部署在通过 CXL 互联的共享内存（DDR/LPDDR）中，多台物理机通过硬件级 Load/Store 指令直接访问，无需经过传统 TCP/IP 网络栈或显式 RPC。
* **软件方案**：**无锁 RCU（Read-Copy-Update）树结构 + GPU 驻留索引**。在单节点内，将轻量化索引树复制一份常驻在 GPU HBM 中，利用 GPU 强大的并行算力，通过简单的 CUDA Kernel 执行百级深度的前缀树并行匹配，将查找时延压制在 $10 \mu\text{s}$ 以内。

### 3. 前缀匹配后的映射与传递（Linkage & Broadcast）
* **现状痛点**：匹配成功后，必须将这些物理 Block 挂载到当前 Engine 的虚拟页表上。如果是多卡张量并行（Tensor Parallelism, TP），每个 GPU 都需要同步更新页表；若 Cache 在非本地卡，还需执行跨卡/跨节点的物理拷贝，导致严重的 PCIe/NVLink 串行等待。
* **硬件增强**：**NVLink Multicast（硬件级组播）与 HBM 直接寻址**。英伟达 Hopper/Blackwell 架构引入了硬件级组播。当某张卡从本地或远端 DDR 拉取到匹配的 KV Cache 块时，通过 NVLink 组播在一个时钟周期内同步推送到 TP 组内的所有 GPU，无需 CPU 参与多次 launch 拷贝。
* **软件方案**：**虚拟化零拷贝映射（Zero-Copy Virtual Linkage）**。对于同节点内的换入（Swapped-In），不改变物理内存位置，仅通过共享内存段或统一内存（UM）的底层的页表指针重映射，让不同 Session 瞬间共享同一段物理 HBM/CXL 内存。

## 二、 补充软件需求规格说明（SRS）

针对上述控制路径的性能退化问题，在先前四层软件架构的基础上，补充以下高价值的正式需求规格（SRS）：

### L1：推理框架层（vLLM / SGLang 侧需求）

#### 需求编号：L1-ENGINE-003
* **组件名称**：前瞻性流式增量索引器（Streaming Incremental Indexer）
* **问题背景**：传统的全量 Prompt 到齐后再计算哈希，会导致在超长文本（Long-Context）场景下，首字时间（TTFT）包含明显的、非必要的哈希计算等待时延。

* **可执行性与验收标准**：
    * 提供配置项 `--enable-streaming-indexing`。
    * 压测 128K 文本输入时，哈希计算本身导致的排队等待时延必须 $\le 1\text{ ms}$。
* **预期性能收益**：大幅消除长文本前置处理真空期，**超长文本场景下的 TTFT 降低 $5\text{-}15\%$**。

### L2：Connector 对接层需求

## 一、 融合底座下的深层技术痛点剖析

### 痛点一：编址与编程模型的“精神分裂”（Duality of Address & Programming Model）
* **第一性原理剖析**：CXL/C2C 期望的是**统一虚拟地址空间（UVA）**，上层直接通过指针解引用或硬件 DMA 搬运数据；而 RDMA/TCP 期望的是**显式的内存缓冲区注册（MR）**，通过 `lkey`、`rkey`、长度和偏移量进行报文组装。如果 L4 层不进行底层抽象，L3 层（传输管理层）就必须写两套完全分叉的调度逻辑，极大增加了系统开销。

### 痛点二：硬件级缓存一致性协议的“内耗血崩”（Coherence Ping-Ponging & Invalidation Storm）
* **第一性原理剖析**：CXL.cache 或某些 C2C 协议默认在硬件层维护全系统的缓存一致性。然而，KV Cache 属于**典型的“写一次，读多次，最后整体销毁”的阶段激活态数据**。如果大批量连续的 KV Cache 块无脑走一致性追踪，会导致海量的跨节点 Cache Invalidate（作废）探测信号瞬间打满总线带宽，引发严重的硬件级“缓存乒乓”，导致正常的算子执行发生内存停顿。

### 痛点三：总线级“硬故障”与网络级“软错误”的隔离鸿沟（Bus MCE vs. Network Timeout）
* **第一性原理剖析**：在 CXL Fabric 拓扑中，如果远端内存节点由于掉电、拔出发生故障，本地 CPU/GPU 硬件在执行 Load/Store 或硬件总线 DMA 时，会触发严重的 **机器检查异常（MCE, Machine Check Exception）**，直接导致整个主板 Linux 内核 Panic 挂起。而在 RDMA/TCP 中，这仅仅表现为一个连接超时（Timeout）事件。如何防止远端节点崩溃导致的连锁机房死机，是融合底座的最大安全隐患。

### 痛点四：同步指令阻塞与异步完成队列（CQ）信号机制的不对称
* **第一性原理剖析**：CXL/C2C 的小块传输依赖于硬件状态机，完成通知是隐式的（数据到齐，指令结束）；RDMA 依赖于显式的完成队列（Completion Queue, CQ）中断或轮询。当 L3 下发一个跨介质混合流任务（一部分走 CXL 内存，一部分走远程 RDMA 磁盘）时，上层控制面调度器很难用统一的事件轮询（Event Loop）来高效感知任务终点。

* **功能需求描述**：

## 一、 不同通信语义适合的 KV Cache 传输场景深度辨析

在统一异构存储池中，KV Cache 的生命周期伴随着不同粒度、不同频次和不同时延敏感度的搬运操作。两种底层语义对应的物理特性，决定了它们在场景适配上的分工：

### 1. 内存语义（Memory-Semantic: CXL.mem / NVLink-C2C）
* **物理特征**：基于硬件级的 `Load/Store` 指令或总线级 Flit 传输。时延在纳秒至亚微秒级（$100\text{ns} \sim 300\text{ns}$），CPU/GPU 硬件 MMU 直接参与寻址，无操作系统内核栈开销，但大规模突发并发写入时容易因硬件流控导致总线拥堵。
* **最适合的 KV Cache 场景**：
    * **元数据与索引树同步**：分布式 Radix Tree 节点的频繁写入、修改、指针跳跃（Pointer Chasing）与跨节点匹配查找。
    * **Decode 阶段的细粒度流式预读/驱逐**：在自回归解码过程中，按需、高频地以极小粒度（如单 Token 或 1-Block 2MB 规模）进行动态水线调配。
    * **同节点/近邻节点多卡（TP/PP）间虚拟共享映射**：多卡之间无需物理拷贝，仅通过硬件级内存指针重定向直接读取公共前缀（System Prompt）。

### 2. 报文/网络语义（Message-Passing/Packet-Semantic: RDMA / TCP）
* **物理特征**：基于异步队列对（Queue Pair, QP）及显式内存注册（MR），采用 `Send/Recv` 或带有 Key 认证的 `Remote Write/Read`。时延在微秒级（$2\mu\text{s} \sim 5\mu\text{s}$），拥有极强的网络级路由扩展性、大吞吐管道流控与硬件级丢包重传容错。
* **最适合的 KV Cache 场景**：
    * **Prefill 结束后的批量异步远端备份**：长文本 Prefill 完成后，将数百 MB 甚至数 GB 的整段 KV Cache 批量、一次性异步推送到 Decode 远端节点或冷存储池。
    * **大规模抢占（Preemption）下的整池换出（Bulk Swapping）**：当集群遭遇突发流量，调度器需要将一整批低优先级 Session 的全量 KV Cache 深度驱逐至远端 DDR 或分布式 NVMe SSD 阵列。
    * **跨机房/跨 Pod 级大跨度拓扑迁移**：超越了 CXL 物理拉线拓扑距离限制的长距离集群调度。

## 二、 架构接口范式抉择：统一语义 vs. 独立语义 vs. 混合范式

针对“向上层提供何种接口语义合适”的问题，从软件工程的**高内聚低耦合**原则与**基础设施极致效能**的双重角度出发，**“两种方式均提供（即：分层治理的混合范式）”是唯一的正确选择。**

### 为什么不能单纯只提供一种？
* **若只提供“抽象统一语义”**：上层推理框架（vLLM/SGLang）和 Connector 确实开发最简单。但是，抽象必然带来信息抹平。上层应用（如 Radix Tree 管理器）拥有明确的业务先验知识（例如：“我知道这段数据是高频访问的系统提示词索引，应该常驻在 CXL 共享内存中”），如果被统一抽象为 `Unified_Copy`，底层的启发式路由引擎只能根据数据大小被动推测，极易发生误判，错失极致性能优化的机会。
* **若只提供“两套不同的独立语义”**：会导致严重的物理拓扑与硬件细节向外泄露。上层应用（如 vLLM 的 `BlockManager`）必须强行写入大量判断底层是 CXL 还是 RDMA 的分支代码，导致 ML 推理框架与底层硬件底座深度锁死，失去了云原生弹性集群的通用性。

### 第一性原理的终极解法：带语义注入的归一化多层抽象（Unified with Semantic Pass-through）
1. **默认路径（90% 场景）**：底座向上层暴露**统一异构描述符接口**（面向对象/逻辑块视图）。框架层无需关心物理介质，由 L3/L4 根据动态探测的拓扑质量、带宽水位、时延和数据块大小，自动运行最优路由。
2. **高级路径（10% 极致性能路径）**：统一接口中允许上层**显式注入“业务先验提示（Application Hints / Quality of Service Flags）”**。如声明 `HINT_LATENCY_CRITICAL`（强制走内存语义通道）或 `HINT_BULK_STREAMING`（强制走报文级 DMA 网络通道）。通过这种方式，既保证了上层代码的整洁度，又赋予了上层应用微调底层硬件行为的终极控制权。

```
+--------------------------------------------------------------------------------+
|                       L1/L2 推理引擎与 Connector 视角                          |
|  调用统一接口：L4_Unified_Transfer(src_token, dst_token, size, QoS_Hint)        |
+--------------------------------------------------------------------------------+
                                        │
             ┌──────────────────────────┴──────────────────────────┐
             ▼ [默认路由：根据数据特征]                              ▼ [显式覆盖：根据 QoS_Hint]
+------------------------------------------------+ +-----------------------------------------------+
|  L3/L4 动态启发式路由引擎 (Heuristic Engine)   | |  业务先验直通通道 (Explicit Pass-through)      |
|  - 小于 64KB -> 自动调度 CXL/C2C 内存通道        | |  - HINT_LATENCY_CRITICAL -> 强制 CXL/C2C     |
|  - 大于 4MB  -> 自动调度 RDMA/GDS 网络通道      | |  - HINT_BULK_STREAMING   -> 强制 RDMA 网络    |
+------------------------------------------------+ +-----------------------------------------------+
```

## 三、 刷新的全局 SRS 需求规格说明书（融合 CXL/C2C 与混合语义）

基于上述分析判定，我们对四层架构的需求文档进行全面重构与补充。以下为刷新后的正式 SRS 文档：

### L1：推理框架层（vLLM / SGLang 侧需求）

#### 需求编号：L1-ENGINE-004
* **组件名称**：业务先验先知 QoS 标签注入器（Application-Aware QoS Hint Injector）
* **问题背景**：框架层拥有关于 KV Cache 语义图谱（如前缀树节点深度、Session 活跃度）的绝对先验知识，但如果无法向下层传递，将导致异构存储池的冷热路由准确率低下。
* **功能需求描述**：
    * 框架层的内存管理器在向 L2/L3 下发卸载/加载任务时，必须在请求上下文描述符中强制编入 `KVCacheQoSHint` 结构体。
    * 标签显式定义三种核心业务特征：
        * `CACHE_PRIORITY_HIGH_LATENCY`：用于系统公共前缀、Radix 根节点，指示底层必须将其保留在内存语义介质中（HBM/CXL.mem）。
        * `CACHE_PRIORITY_BULK_THROUGHPUT`：用于 Prefill 刚结束的大块换出或长文本 Swapping，指示底层启动大吞吐网络搬运。
        * `CACHE_PRIORITY_COLD_ARCHIVE`：用于长时未访问的会话，指示底层直接推向 SSD 磁盘介质。
* **可执行性与验收标准**：
    * 推理引擎对底层的 API 调用接口全面升级，增加参数 `const KVCacheQoSHint& hint`。
* **预期性能收益**：底层异构介质的冷热数据分类准确率提升至 **$98\%$ 以上**，大幅减少因误判导致的介质二次颠簸时延。

### L3：传输管理层需求

#### 需求编号：L3-MGMT-004（重大重构刷新）
* **组件名称**：混合双模自适应协议路由与桥接引擎（Hybrid Dual-Mode Protocol Routing Engine）
* **问题背景**：统一存储集群需要同时承载内存语义总线与报文网络总线，上层需要接口归一，但不能失去对底层硬件的针对性调配。
* **功能需求描述**：
    * L3 路由引擎必须实现**双模控制逻辑**：
        1. **隐式自适应模式（Implicit Mode）**：当上层传入的 `QoS_Hint` 为默认值时，路由引擎根据动态探测到的硬件拓扑、链路当前带宽水位以及单次数据块（Chunk）大小进行物理选路。若 `Size $\le 128 \text{ KB}$` 且存在 CXL/C2C 连通路径，直接分流至内存语义驱动；若 `Size $> 2 \text{ MB}$`，自动切片并分流至 RDMA 网络。
        2. **显式强制覆盖模式（Explicit Pass-through Mode）**：当上层传入的 `QoS_Hint` 带有强制标记（如 `HIGH_LATENCY`），路由引擎必须豁免内部大小判断，直接开辟绿色通道，强行调用底层的 CXL/C2C 硬件直通接口， bypass 掉任何网络组包和内存注册逻辑。
* **可执行性与验收标准**：
    * 编写路由单元测试，构造高并发混合大小 KV 块，验证在不同 QoS 标签注入下，底层的分流准确率达到 100%。
* **预期性能收益**：在接口统一的前提下，元数据控制流操作延迟锁定在 **$\le 300\text{ ns}$**，大块数据吞吐打满 800Gbps 网络线速，实现“一赋双能”。

### L4：底层传输层需求

#### 需求编号：L4-TRANS-003（重构刷新）
* **组件名称**：多协议对称驱动虚拟化核心（Symmetric Multi-Protocol Driver Core）
* **问题背景**：需要向下对接高度分化的 CXL 总线驱动与 RDMA 网卡驱动，必须在 L4 提供一个高度归一化的对称传输层，以支持 L3 的双模路由下发。
* **功能需求描述**：
    * L4 核心必须抽象并暴露一套通用的**统一虚拟数据描述符（Unified Virtual Descriptor, UVD）**。
    * 提供两类底层驱动后端的对称绑定：
        * **后端的内存语义通道（Memory-Semantic Backend）**：封装 CXL.mem 地址空间映射函数，通过硬件 DMA 或直接 `memcpy`（Write-Combining 模式）读写映射的主机/远端 DDR。
        * **后端的报文网络通道（Packet-Semantic Backend）**：封装 IBV (Infiniband Verbs) 驱动，维护异步队列对（QP），调用 `ibv_post_send` 和 `ibv_post_recv` 执行远程直接内存访问。
* **可执行性与验收标准**：
    * L4 必须向 L3 暴露两个具有相同函数签名的底层传输 API：`L4_Direct_Memory_Write_Async()` 与 `L4_Message_Network_Post_Async()`。
* **预期性能收益**：为 L3 层提供完美的软硬件解耦底座，彻底消除了框架层维护多套底层网络栈的混乱局面，**底层驱动维护成本下降 $60\%$**。

#### 需求编号：L4-TRANS-007（新增需求）
* **组件名称**：CXL/C2C 零拷贝内存语义通道扩展（CXL/C2C Zero-Copy Memory Channel）
* **问题背景**：细粒度、低延迟的 KV Cache 匹配和频繁换入换出场景下，必须有极致的、无任何中间层开销的物理通道保障。
* **功能需求描述**：
    * 专门针对 `CACHE_PRIORITY_HIGH_LATENCY` 场景，提供专门的内存语义低延迟直通硬件通道。
    * 该通道在底层必须支持**零拷贝指针穿透技术**。当上层触发加载时，若探测到目标前缀 KV Cache 已经在本地机型的 CXL 扩展共享内存池中，底层传输层不启动任何数据拷贝动作，而是直接将该 CXL 物理内存页的虚实映射关系（Page Table Entries）直接插入到目标 GPU 的虚拟地址空间（UVM）中，实现硬件级的“无感即时就位”。
* **可执行性与验收标准**：
    * 通过监控数据搬运流量，在命中 CXL 内存池的前缀加载测试中，**系统物理总线拷贝流量（Bus Copy Traffic）实测为 0**。
* **预期性能收益**：同节点/近邻节点下的前缀匹配加载时延从微秒级直接抹平至**零延迟（仅 $\le 100\text{ ns}$ 的页表挂载耗时），彻底释放 RadixTree 前缀复用的极限潜力**。

#### 需求编号：L4-TRANS-008（新增需求）
* **组件名称**：RDMA 大吞吐网络异步流控管道（RDMA Bulk-Streaming Async Pipeline）
* **问题背景**：长文本全量驱逐时，如果大块数据长时间占用总线，会阻塞轻量级内存语义通道的正常握手，引发控制流与数据流的严重相互踩踏。
